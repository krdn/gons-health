# Harness 전달 설계안: tool 입력 JSON 깨진 `\u` 이스케이프 대응

작성일: 2026-06-21
대상: Ultracode / Anthropic harness 팀
관련: `docs/diagnosis-tool-format-error-20260621.md` (근본 원인 진단)

---

## 0. TL;DR

tool 인자 JSON에서 한글을 `\uXXXX`로 직렬화하는 도중, `\u` 뒤 hex 4자리를 다 채우지 못하고
임의 문자가 섞여 들어가 JSON 파싱 전체가 실패한다(`InputValidationError: could not be parsed as JSON`).

- **이건 복구로 풀 문제가 아니다.** 관찰된 변종 일부는 의도한 문자를 **재구성할 수 없다**(아래 §2).
- **근본책은 예방**: tool 인자 생성 시 JSON 문법을 토큰 레벨로 강제하는 constrained/grammar-guided decoding.
- 복구는 "owning 못 하는 버그를 막는 catch-net"으로만, **좁게 + 애매하면 시끄럽게 실패**.

권장 우선순위: **①예방 → ②교정 재시도 → ③(선택)좁은 복구 → ④그 외 fail-loud**

---

## 1. 증거 레이어 (먼저 확정할 것 — 틀리면 fix가 틀림)

corruption이 **어느 레이어**에서 생기는지가 fix를 가른다. 현재까지의 단서:

- 분석 대상은 transcript의 `input.__unparsedToolInput.raw` 문자열.
  `.raw`는 **harness가 파싱하기 직전에 받은 입력** → corruption은 harness 로깅이 아니라
  **상류(모델 생성 단계 또는 API 직렬화 단계)**에서 이미 발생해 있음.
- 같은 세션에서 `ScheduleWakeup`은 한글을 **리터럴 UTF-8**로 출력해 9/9 성공,
  `AskUserQuestion`은 `\uXXXX` 이스케이프로 출력하다 실패.
  → **모델 측 직렬화 선택 차이일 확률이 높다** (단정 금지).

**harness 팀 확인 요청 (와이어 바이트 기준):**
1. 모델이 `\u` 토큰을 뱉다가 hex 자리에서 미끄러진 것인가? (생성 단계 결함)
2. 아니면 API/SDK가 리터럴 UTF-8을 `\u`로 재인코딩하다 잘린 것인가? (직렬화 단계 결함)

→ 1번이면 §3(constrained decoding)이 근본책. 2번이면 직렬화 파이프라인 수정이 근본책.
   **원인 레이어가 다르면 처방이 다르므로 이 확인이 선행되어야 한다.**

---

## 2. 결함 클래스 (관찰된 변종 — 복구 가능/불가 판정)

실제 transcript(2026-06-21, AskUserQuestion 6건 실패)에서 수집:

| 깨진 형태 | 구조 | 의도 문자 복구 | 비고 |
|---|---|---|---|
| `\ub뢰` | `\u`+hex 1자리 + **리터럴 한글** | **부분 가능** | '뢰'가 그 자리에 리터럴로 존재 (3회 관측, 최빈) |
| `\ubt`  | `\u`+hex 1자리 + **ASCII 문자** | **불가** | 의도 문자 정보 소실 |
| `\ud ` | `\u`+hex 1자리 + **공백** | **불가** | 의도 문자 정보 소실 |
| `\uc`   | `\u`+hex 0자리 + 다음 글자 | **불가** | hex 시작도 못 함 |

**클래스 정의:** `\u` 직후 hex가 0~3자리만 오고, 4자리를 채우기 전에 **non-hex 문자**(한글/ASCII/공백 등)가 등장.

**핵심 결론:** 변종 4개 중 3개는 의도 문자를 **재구성할 수 없다**.
즉 "모든 깨진 `\u`를 복구한다"는 설계는 불가능하며, 추측 복구는 **틀린 데이터를 silent하게 생성**한다.
이 클래스는 open-ended 생성 glitch라 패턴 열거로 가둘 수 없다 → 예방 + fail-safe로 대응한다.

## 3. ① 예방 (근본책, 헤드라인)

### 3.1 Constrained / grammar-guided decoding
tool 인자 생성 시 출력 토큰을 **JSON 문법으로 토큰 레벨 제약**한다.
`\u` 이스케이프 상태에 진입하면 다음 토큰을 **hex 4자리로만** 허용 → 깨진 `\u`가 **애초에 생성 불가**.

- 가장 깨끗한 해법. corruption이 생성 단계(§1-1)일 때 근본책.
- 라이브러리 예: JSON Schema 기반 constrained decoding (outlines, llguidance, xgrammar 류).

### 3.2 직렬화 단계 정규화 (corruption이 §1-2일 때)
API/SDK가 tool 인자를 직렬화할 때 **리터럴 UTF-8을 우선**하고 ASCII-only 이스케이프 강제를 끈다.
ScheduleWakeup이 리터럴로 통과한 경로가 이미 안전함을 보여줌.

---

## 4. ② 교정 재시도 (예방이 불가/부분적일 때)

현재 harness는 파싱 실패 시 `"Retry with valid JSON"`을 띄운다. 그러나 데이터-라이선스 질문은
**1910 → 1943 → 1858 bytes로 3연속 재시도 실패**했다. "그냥 재시도"는 같은 결함을 재생산한다.

개선된 재시도:
1. 재시도 프롬프트에 **교정 지시 주입**: "한글 등 비-ASCII 문자를 `\uXXXX`로 이스케이프하지 말고
   리터럴 UTF-8로 출력하라."
2. 가능하면 **재시도 시 constrained decoding ON** (§3.1).
3. 입력이 큰 도구(AskUserQuestion 등)는 재시도 시 **텍스트 축소 유도** → 이스케이프 토큰 수↓.

---

## 5. ③ 좁은 복구 (선택 — 잔여위험 수용 시만)

§2에서 보듯 복구는 한 변종(`\u`+부분hex + non-ASCII 리터럴)에만 적용 가능하다.

### 두 갈래 — harness 팀이 선택
- **A안 (무복구, 권장)**: 깨진 `\u`는 전부 reject + §4 재시도. 데이터 손상 위험 **0**.
  약물 안전 도구를 함께 구동하는 환경이므로 기본값으로 권장.
- **B안 (좁은 휴리스틱, 잔여위험 수용 시)**:
  ```
  IF  '\u' + hex{0,3} + (다음 문자가 non-ASCII 리터럴: 코드포인트 ≥ 0x80) 발견:
      → '\u' + 부분hex 토막을 폐기하고, 그 리터럴 문자를 그대로 살린다  (\ub뢰 → 뢰)
  ELSE (다음 문자가 ASCII/공백/제어 등):
      → 복구 불가. reject + 재시도.   ← 절대 추측 복구 금지
  ```
  주의: B안도 휴리스틱이며 잔여 오복구 위험이 있다. "의심스러우면 복구하지 말고 reject".

### 1원칙
**애매하면 시끄럽게 실패(fail-loud)한다. silent 추측 복구는 금지.**

---

## 6. ④ 그 외 — Fail-loud

§3~§5로 처리되지 않는 모든 깨진 입력은 **명시적 에러 + 재시도**로 처리한다.
절대 부분 파싱 결과를 그대로 통과시키지 않는다 (silent partial 금지).

---

## 7. 수용 기준 (검증 방법)

1. **재현 픽스처**: §2의 4개 변종을 담은 깨진 JSON 문자열을 입력으로 주입.
2. 예방(§3) 적용 시: 깨진 `\u`가 생성/직렬화 단계에서 발생하지 않음.
3. 복구 A안: 4개 변종 모두 reject + 재시도 → 재시도 성공.
4. 복구 B안: `\ub뢰`만 `뢰`로 복구, 나머지 3개는 reject. **오복구(틀린 글자 생성) 0건**.
5. 재시도(§4): 교정 지시 주입 후 **3연속 실패 → 1회 성공**으로 개선되는지 회귀 테스트.

## 8. 한계 / 정직한 명시

- 이 설계는 transcript 표본(AskUserQuestion 6건, 변종 4종)에 근거한다.
  생성 glitch는 open-ended라 표본이 클래스 전체를 대표하지 못한다 → 그래서 예방+fail-safe 중심.
- corruption의 정확한 레이어(§1)는 harness 팀의 와이어 바이트 확인 전까지 **미확정**.
  레이어가 다르면 §3의 근본책(3.1 vs 3.2)이 달라진다.
- 모델 측 행동(리터럴 출력)은 부분적으로 stochastic이라 **빈도만 낮추고 0은 불가** →
  근본 0은 harness 레이어(§3)에서만 달성 가능.
